# MilovanaTeaseEditor — Functional Description (UI Redesign Brief)

> **For:** Claude Design.
> **Purpose:** Describe *what the tool does* (functionality and actions) so a new user interface can
> be designed. This is **not** a layout spec — it deliberately avoids prescribing screens, panels,
> colours, or components. Where file or JSON detail appears, it is only because it drives what the UI
> must show or let the user do (image previews, tag fields, asset resolution, workflow state).

---

## 1. Purpose & audience

The **MilovanaTeaseEditor** is an authoring environment for **Milovana EOS teases** — interactive,
page-based scripts (stored as a `tease.json` file) that combine text, images, audio, timers, and
branching choices. Today the shipped tool (the *MilovanaEosEditor*) covers only a narrow slice of the
authoring pipeline: mapping local image/audio files to Milovana's server-side IDs, and tagging images
with descriptive metadata. The redesign should keep that functionality but reframe the whole tool
around the **complete authoring workflow**, and be structured to absorb three planned extensions
(tease browser, script editor, workflow dashboard — see §6–§8). One stance to note early: the **script
editor is text-first** — the author edits the `script.md` source directly in a rich, IDE-style editor
with inline enhancements (image previews, rendered formatting, autocomplete, live validation), *not* a
forms/builder UI with a control per action (see §7).

**Who uses it:** a single author, working alongside Claude. Claude reads images and proposes tags,
writes/edits the script, and runs automated steps; the human author reviews, verifies, and makes
creative decisions. The UI must therefore be comfortable both for **manual editing** and for showing
**the results of automated steps** clearly.

### Platform (call this out in mockups)

This is a **functional desktop application** for **desktop use**. The target window is **landscape,
16:9, and resizable**. Mockups should assume a resizable desktop window that **reflows gracefully**
across sizes — wide and roomy at full screen, still usable when the window is made smaller. It is
**not** a mobile or fixed-width layout. Density matters: this is a working tool an author keeps open,
not a marketing page.

## 2. Domain primer (just enough to design against)

A **tease** is a self-contained project. Everything for one tease lives in a single folder, and the
files in that folder are the backbone the UI organises itself around:

| File / folder | What it is | Who maintains it |
|---|---|---|
| `script.md` | **The human-editable source of truth** for the tease — the script, written page by page in a simple marker language. | Author + Claude |
| `tease.json` | **The final, playable tease**, generated from the script. This is what gets uploaded to Milovana. | Generated (don't hand-edit) |
| `asset-map.json` | A generated index that maps each **local image/audio file** to its **Milovana locator** (the address used inside `tease.json`). | Generated by the tool |
| `asset-content.json` | **Descriptive tags** for each image (what it shows, how intense it is). Drives image selection and previews. | Claude (proposes) + author (verifies) |
| `Gallery/<bucket>/` | Local image files, organised into **buckets** (one subfolder per bucket). | Author provides |
| `Files/` | Local audio files (e.g. metronome loops). | Author provides |

### Themed gallery buckets (an important concept for the UI)

Images are organised into **themed buckets**, where **one bucket ≈ one mood/pace of the script**
(e.g. `solo-sensual`, `machine-soft`, `machine-hard`, `climax`). This matters for design because:

- The script can say "show *any* image from the `machine-hard` bucket," and it stays on-theme
  automatically — so the UI should treat **bucket** as a first-class way to browse and pick images.
- Hard rules ("this section must only show the model alone") become **structural**: a section simply
  draws from a `solo-*` bucket. The UI should make a bucket's theme/intent visible at a glance.

### Two independent tagging axes (matters for previews and selection)

Each image carries two **separate** descriptive axes that the UI must keep distinct:

- **Pace (1–5)** — energy/tempo *only*, independent of nudity. Anchored to musical tempo bands so it
  can be matched to a metronome BPM (1 = still/posed … 5 = frantic/hard).
- **Explicitness** — how much is shown (`clothed` → `underwear` → `partial-nudity` → `nude` →
  `explicit`), independent of pace.

The point of keeping them separate: a tease's **tempo arc** and its **exposure arc** escalate
independently. The UI should let the author see and filter on both axes without conflating them.

## 3. The authoring workflow — the spine of the new UI

The whole tool exists to move a tease through an ordered pipeline. The single most important design
goal is that **the author always knows what's done and what to do next**, and can act on it. Each step
below notes **who** does it, whether it's **one-time or repeatable** (and what triggers a re-run), the
**signal that it's done**, and **the next action**.

| # | Step | Who | One-time / repeatable | "Done" signal | Next action |
|---|---|---|---|---|---|
| 1 | **Create the tease** — make the tease folder with the right structure (`Gallery/`, `Files/`). | Author (Tool can scaffold — see §6) | One-time | Folder exists | Plan assets |
| 2 | **Write the script** — author the script page by page in `script.md`. | Author + Claude | **Repeatable** (edited continuously) | `script.md` has content | Plan/refine assets |
| 3 | **Plan assets** — decide the themed buckets and the audio files needed. | Claude (proposes), author approves | One-time (re-do if scope grows) | Bucket list agreed | Set up & upload assets |
| 4 | **Set up & upload assets** — create the bucket folders and `Files/`, drop in the real image/audio files, and upload matching galleries to the Milovana editor. | Author | **Repeatable** (whenever new images/audio are added) | Files present locally + uploaded | Export the manifest |
| 5 | **Export the manifest** — download the stub `tease.json` from the Milovana editor (it carries the server's gallery/file IDs and hashes). | Author | **Repeatable** (re-export after new uploads) | `tease.json` manifest present | Generate the asset map |
| 6 | **Generate the asset map** — the tool joins the manifest to the local files and writes `asset-map.json`; it also **seeds empty tag rows** for any newly-found images. | **Tool** (Claude/author triggers it) | **Repeatable & idempotent** — safe to re-run after every asset change; only adds what's new | `asset-map.json` up to date; all files matched | Tag the new images |
| 7 | **Tag the images** — view each (untagged) image and fill in its descriptive tags in `asset-content.json`. | Claude (proposes), author verifies | **Repeatable** — only the new/untagged images need attention | Every image has tags | Verify |
| 8 | **Verify tags** — review and adjust the tags. | Author | **Repeatable** | Author satisfied | Generate the tease |
| 9 | **Generate the tease** — resolve each image/audio reference in the script to a locator and build the final `tease.json`. | Tool (Claude/author triggers it) | **Repeatable** — re-run after any script or asset change | `tease.json` regenerated, all references resolve | Upload & test |
| 10 | **Upload & test** — upload `tease.json` to Milovana and play through it. | Author | **Repeatable** | Plays correctly | Iterate or finish |
| 11 | **Iterate** — refine on feedback; loop back to step 2, 4, or 7 as needed. | Author + Claude | **Repeatable** | — | — |

**Design implications:**

- Several steps are **repeatable**, and the common trigger is *"new images were added."* The UI must
  make re-running (esp. steps 6, 7, 9) frictionless and must show clearly that re-running is **safe**
  (it only processes what's new / regenerates derived files).
- "Done-ness" is **derived from the files on disk** (does `asset-map.json` exist? do all images have
  tags? does every reference in the script resolve?). The UI computes and displays this state rather
  than relying on the author to remember.
- Steps 6 and 9 are **automations** the tool runs and reports on; everything else is human/Claude
  editing the tool should facilitate.

## 4. Current functionality (what exists today)

These are the capabilities the tool already has, described as **user-facing actions**, not as the
current single-window layout. The redesign must preserve all of them.

### 4.1 Open / locate a tease

- **Open a tease's content** for editing (today this means opening its `asset-content.json`). The tool
  can also **auto-locate a default tease** on startup so the author lands somewhere useful.
- *Redesign note:* this is the seed of the **tease browser** (§6) — selecting *which* tease to work on
  should become a first-class action, with the current "open a file" reduced to an internal detail.

### 4.2 Browse galleries and images

- **Browse the tease's galleries** and, within each, its **images** — a navigable hierarchy of
  *gallery → images*.
- **Select a gallery** to see its details; **select an image** to view and tag it.

### 4.3 Edit gallery metadata

For a selected gallery, the author can edit:

- **Theme** — the gallery's mood/pace category (e.g. `solo-sensual`, `machine-hard`).
- **Theme note** — a free-form note describing the gallery's intent or constraints.

These are how the "themed bucket" idea (§2) is recorded, so they should be prominent and easy to edit.

### 4.4 View an image

- **Preview the selected image** at a usable size.
- If the image file is missing on disk, the tool shows a clear **"image file not found"** state rather
  than a broken preview. (This not-found state is a recurring requirement — see §5.)

### 4.5 Tag an image

The core manual task. For each image the author sets:

| Field | Meaning | Input style |
|---|---|---|
| **Subject** | Who/what is shown: `solo` / `machine` / `partner`. Used to enforce constraints (e.g. tutorial sections = `solo` only). | Suggested values + free text |
| **Pace (1–5)** | Energy/tempo only, **ignoring** nudity. BPM-banded so it can be matched to a metronome. | Numeric 1–5 |
| **Explicitness** | How much is shown: `clothed` / `underwear` / `partial-nudity` / `nude` / `explicit`. | Suggested values + free text |
| **Orientation** | `portrait` / `landscape`. Usually pre-filled automatically from the image's real dimensions; editable. | Suggested values |
| **Notes** | Free-text pose/setting cue for a specific beat (e.g. "standing, side profile, daylight window, looking to camera; still/posed"). | Free text |

Design notes:
- The suggested values come from a **vocabulary** stored with the tease, so dropdowns should be
  populated from data, not hard-coded, and should still allow free text.
- **Pace and explicitness are independent axes** (§2) and must not be visually merged.
- A compact **tag summary** (e.g. *subject · pace · explicitness*) should be visible when browsing
  many images, so the author can scan a gallery's coverage without opening each image.

### 4.6 Save & unsaved-change safety

- **Save** writes the author's edits back to the tease's content.
- The tool **tracks unsaved changes** (a "dirty" indicator) and **prompts before discarding them**
  when navigating away or closing. The redesign must keep this safety behaviour everywhere edits
  happen.

### 4.7 Generate the asset map (the key automation)

A single action that:

- **Joins** the exported manifest (`tease.json`) to the **local files** and writes `asset-map.json`,
  producing a ready-to-use **locator** for every image and audio file.
- **Seeds empty tag rows** for any images that don't yet have tags — so the tagging step (4.5) only
  has to deal with what's new.
- **Reports the result**: how many galleries and files were processed, how many images **matched**,
  and — importantly — **what didn't match** (manifest entries with no local file, local files not in
  the manifest, unmatched audio). This report is a primary surface the UI must present well, because
  unmatched assets are the author's cue to fix something.

Two properties the UI should lean on:

- It is **safe to re-run** at any time and **idempotent** — re-running with no new assets changes
  nothing; re-running after adding images only adds the new ones. This is the canonical **repeatable**
  step, and the UI should encourage running it whenever assets change.
- It needs **no network and no AI** — it's a local, deterministic file operation.

### 4.8 Headless / scriptable run

The asset-map generation can also be run **without opening the GUI** (a batch/automation path used in
build pipelines). The redesign doesn't need to expose a terminal, but it should keep this action
**one click away** in the UI, since it's the same operation the author will run repeatedly.

## 5. How images & assets resolve (drives previews & selection)

The UI's previews and image-picking depend on understanding how a *reference in the script* becomes
*a real image on screen*. The design only needs the shape of this, not the byte-level detail:

- **Images resolve by content (SHA-1 hash), audio by filename.** A local image file is matched to its
  Milovana ID by hashing its bytes; audio is matched by its filename. The UI should **never** rely on
  reported image width/height for identity (those are unreliable) — it relies on the **asset map**.
- **Image *selection* joins two files on the filename:** the **tags** (`asset-content.json`, "what
  this image is") and the **locators** (`asset-map.json`, "how to address it in the tease"). This join
  is exactly what lets a future **script editor preview and pick images by meaning** — e.g. "show a
  `nude`, `pace 4` image from the `machine-hard` bucket." Design the image picker around *tags +
  bucket*, with the locator handled invisibly underneath.
- **Unresolved / missing assets must be surfaced.** Three failure shapes the UI must represent clearly:
  1. an image referenced by the script that **has no locator** (asset map out of date → prompt re-run
     of step 6);
  2. a local image file that is **missing on disk** (the "file not found" preview state);
  3. an image with **no tags yet** (needs the tagging step).
  Each should read as an actionable problem tied to a specific workflow step, not a silent failure.

## 6. Future extension 1 — Tease browser

A way to choose which tease to work on, and to start new ones. **The tool must let the author:**

- **See all existing teases** in the workspace, each with an at-a-glance **workflow status** — how far
  through the pipeline (§3) it is (e.g. "assets mapped, 12 of 40 images tagged, not yet generated").
  This status is derived from which files exist and how complete they are.
- **Open a tease**, which makes it the **active tease** for every other area of the app (§9).
- **Create a new tease**, which **scaffolds the correct folder structure** automatically — the tease
  folder, an empty `Gallery/` and `Files/`, and whatever starter files the workflow expects — so the
  author starts at step 2/3 cleanly rather than hand-building folders.
- **Recognise incomplete or partial teases** (e.g. assets present but never mapped, or a manifest with
  no script yet) and present that honestly in the status, so a half-finished tease isn't mistaken for
  a finished one.

Constraints Design should account for: tease names become folder names (so a **naming rule / valid-name
check** is needed on creation), and the browser should handle an **empty workspace** (no teases yet)
with a clear path to "create your first tease."

## 7. Future extension 2 — Script editor

Where the author actually writes and edits the tease.

**This is a text-first editor, not a forms-based one.** The author edits the `script.md` file
*directly* — the script is written in a simple marker language (the script DSL), and the editor should
feel like editing that text, the way a code editor feels. **Do not** introduce a separate UI control
(panel, form, dialog) for each action type; the action *is* its line(s) of script text. The value the
tool adds is a layer of **inline enhancements on top of the text** — think "rich code editor for the
script DSL," in the spirit of a modern IDE (e.g. Rider/VS Code) rather than a drag-and-drop builder.

The text the author edits is a sequence of **pages**, each a list of **actions** written as markers
(show text, show image, wait/timer, play audio, go to a page, choice, notification, end). Design
doesn't need the exact syntax — it needs to support editing that text fluently and enriching it.

**The editor must provide these enhancements over the raw text:**

- **Syntax-aware editing of the DSL** — highlight markers, parameters, and payloads; make the script's
  structure (page boundaries, action types) visually legible while it's still fundamentally text.
- **Autocomplete / IntelliSense-style assistance** (Rider-style) while typing — suggest action
  markers, valid parameter names, and valid parameter values; complete things like image
  bucket/filenames and audio filenames from the actual assets, and page names for jump targets.
- **Inline image preview & picking.** Where a line references an image, show a **thumbnail/preview
  inline** (resolved through the asset map + tags, §5), and let the author **pick or swap** the image
  by browsing tagged galleries/buckets filtered on the tag axes (subject, pace, explicitness) — i.e.
  choose by *meaning*. Handle the **"random from a bucket"** case by communicating that the image is
  *one of several* on-theme options, not a fixed pick (e.g. show representative thumbnails of the
  pool).
- **Render formatting instead of markup.** Show **actual text colour and alignment/facing** in the
  editor rather than raw `<span>`/style markup — the author sees the styled result while still editing
  text, and isn't hand-writing HTML spans.
- **WYSIWYG-style `say` preview.** Let the author **preview a full text ("say") action the way it will
  appear in Milovana** — matching the player's layout/typography — so they can judge the on-screen
  result without uploading.
- **Editing helpers / live validation as they type**, surfaced inline at the relevant line:
  - **Verify "go to" targets** — referenced page names must exist; flag broken/typo'd jumps (and ideally
    offer to jump to, or create, the target page).
  - **Validate action parameters** — known parameter names, sensible value ranges/types, required
    fields present.
  - **Flag unresolved assets** — an image/audio reference with no locator yet (asset map out of date,
    §5), or a local file missing on disk.
  - Other structural checks: a missing required entry page, references to undefined pages, etc.
- **Easy navigation of a long script** — e.g. jump between pages / an outline of page names — without
  abandoning the single editable text surface.

**Export to `tease.json`** — a clear action that **builds the final playable tease** from the script.
After export, the UI reports success and **flags anything that didn't resolve** (unresolved
image/audio references, validation problems) so the author can fix them before uploading.

Cross-reference: the existing **script DSL and page/action model** are documented in
`milovana/Documentation/EOS-Tease-Authoring-Guide.md`; Design should skim it to understand the marker
shapes the editor is enriching.

## 8. Future extension 3 — Workflow & automation dashboard

The "home base" that makes the pipeline (§3) visible and actionable. **The tool must:**

- **Show the full workflow with clear state per step** — **done / current / next** — for the active
  tease, derived from the files on disk (not from the author's memory).
- **Let the author execute the next step in one action** wherever the tool can do it (notably the two
  automations: *generate asset map* and *generate tease*), or clearly point to the manual action
  needed (e.g. "export the manifest from the Milovana editor") when it's something only the author can
  do outside the tool.
- **Clearly mark repeatable steps and offer re-runs.** The dashboard is where the author returns after
  adding new images: it should make "**regenerate the asset map**" and "**tag the new images**" and
  "**regenerate the tease**" obvious, and convey that re-running is **safe and incremental**.
- **Show the output of automated steps** — especially the asset-map **match report** (matched vs.
  unmatched assets) and the tease-generation result — and turn any **blocking problems** into clear,
  actionable items linked back to the step (or the page) that needs attention.

The dashboard and the other two areas share one **active tease**; acting on a step here should take the
author straight into the relevant editor (e.g. "tag new images" → the image tagging view filtered to
untagged images).

## 9. Cross-cutting UX requirements

- **Desktop, landscape 16:9, resizable.** Layout must reflow across window sizes — comfortable at full
  screen, still usable when smaller. Not fixed-width, not mobile.
- **One persistent "active tease" context** shared by the browser, the script editor, the tagging
  views, and the workflow dashboard. Switching teases switches everything consistently, and the
  current tease should always be identifiable.
- **State derived from reality.** Workflow status, "done/next," and problem indicators are computed
  from which files exist and how complete they are — the UI reflects the actual state of the tease,
  not a separate stored flag the author has to maintain.
- **Unsaved-change safety everywhere** edits happen (tags, gallery metadata, script): a dirty
  indicator plus a confirm-before-discard prompt.
- **First-class empty / error / not-found states**, including: no tease selected, empty workspace (no
  teases), image file missing on disk, image not yet tagged, and a script reference whose locator
  doesn't resolve. Each should read as an actionable problem pointing at the workflow step that fixes
  it (§5).
- **Offline & local.** The tool itself makes **no network or AI calls** — tagging input and creative
  decisions come from outside (the author and Claude). The UI should never imply it's "thinking" or
  contacting a service; its automations are fast, local, deterministic file operations.

## 10. Glossary

| Term | Meaning |
|---|---|
| **Tease** | One interactive, page-based Milovana project; lives in its own folder. |
| **Page** | A named section of a tease; holds an ordered list of actions. |
| **Action** | One step on a page — show text, show image, wait, play audio, go to a page, choice, notification, end. |
| **Gallery / bucket** | A themed folder of images, ideally matching one mood/pace of the script (e.g. `machine-hard`). |
| **Locator** | The address the tease uses to reference an image (`gallery:…`) or audio file (`file:…`). |
| **Manifest** | The gallery/file IDs and hashes exported from the Milovana editor (inside the stub `tease.json`). |
| **Asset map** (`asset-map.json`) | Generated index linking each local file to its Milovana locator. |
| **Asset content** (`asset-content.json`) | The descriptive **tags** for each image. |
| **Pace** | An image's energy/tempo (1–5), independent of nudity; matchable to a metronome BPM. |
| **Explicitness** | How much an image shows, independent of pace. |
| **Script** (`script.md`) | The human-editable source the playable `tease.json` is generated from. |
